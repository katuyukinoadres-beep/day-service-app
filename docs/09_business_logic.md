## 1. 概要

## 2. ドメイン領域

### 2.1 認証・ユーザー管理

#### 概要

ユーザーの認証と権限管理を担うドメイン。メインアプリではSupabase Authによるメール・パスワード認証を使用し、管理画面では独立した共有パスワード認証を採用する。新規登録時には施設の作成も同時に行い、施設なしのユーザーは存在しない。

#### フロー図

**新規登録フロー（施設オーナー）:**

```
[入力フォーム] → [Supabase Auth signUp] → [自動signIn]
                                              ↓
                                    [facilities INSERT]
                                              ↓
                                    [profiles INSERT (role="admin")]
                                              ↓
                                    [ホーム画面へリダイレクト]
```

**スタッフ招待フロー:**

```
[admin: 招待リンク生成] → [リンク共有: {origin}/join?facility={facilityId}]
                                              ↓
                          [招待者: リンクを開く → 入力フォーム]
                                              ↓
                          [Supabase Auth signUp → 自動signIn]
                                              ↓
                          [profiles INSERT (role="staff", facility_id=URLパラメータ)]
                                              ↓
                          [ホーム画面へリダイレクト]
```

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| AUTH-R001 | 新規登録時の施設作成 | 新規登録時は必ず施設を作成する。施設なしのユーザーは存在しない |
| AUTH-R002 | 初回ユーザーのロール | 新規登録（施設作成）したユーザーは必ず `admin` ロール |
| AUTH-R003 | 招待経由のロール | 招待リンク経由のユーザーは必ず `role = "staff"` |
| AUTH-R004 | 招待リンクの施設ID | `facility_id` はURLパラメータから取得（施設選択UIなし） |
| AUTH-R005 | 自己ロール変更の禁止 | ユーザーは自分自身のロールを変更できない（UIで無効化） |
| AUTH-R006 | 管理画面フォールバック | `ADMIN_PASSWORD` 未設定時のフォールバック値は `"Admin"` |
| AUTH-R007 | 管理画面セッション | Cookie `admin_session` の値は `"authenticated"`、有効期限24時間 |

**ロール変更マトリクス:**

| 操作 | 実行者 | 対象 |
|---|---|---|
| `staff` → `admin` | `admin` | 同施設の他スタッフ |
| `admin` → `staff` | `admin` | 同施設の他スタッフ |
| 自分のロール変更 | -- | 不可（UIで無効化） |

#### エッジケース

| ケース | 発生条件 | 処理 |
|--------|---------|------|
| 招待リンクの施設IDが無効 | 存在しない `facility_id` がURLに指定された場合 | RLSポリシーにより `profiles` のINSERTは成功するが、該当施設のデータにアクセスできない |
| 管理画面のPW未設定 | 環境変数 `ADMIN_PASSWORD` が未設定 | フォールバック値 `"Admin"` でログイン可能（AUTH-R006） |

→ 関連: 08_security.md §2（認証セキュリティ）

### 2.2 マルチテナント分離

#### 概要

全データを `facility_id` で施設単位に分離し、RLS（Row Level Security）によりユーザーが自施設のデータのみアクセスできることを保証する。スーパー管理者は全施設のデータに横断アクセス可能。

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| MT-R001 | 施設ID必須 | 全データテーブルに `facility_id` カラムを設置 |
| MT-R002 | RLS適用 | 全テーブルでRLSを有効化し、`profiles.facility_id` と一致するレコードのみアクセス可能 |
| MT-R003 | スーパー管理者 | `profiles.is_super_admin = true` のユーザーは全施設のSELECT/UPDATEが許可される |
| MT-R004 | 管理画面のRLSバイパス | 管理画面APIは `service_role` キーを使用してRLSをバイパスし、全施設データにアクセスする |

→ 関連: 03_DB設計書.md、08_security.md §4.2

### 2.3 児童管理

#### 概要

施設に所属する児童の登録・編集・論理削除を管理するドメイン。児童には支援領域タグと個別目標を設定でき、これらはフレーズ表示の優先順位やAI記録生成の入力データとして使用される。

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| CHILD-R001 | 必須フィールド | `name`（氏名）と `icon_color` のみ必須 |
| CHILD-R002 | アイコンカラーデフォルト | 未指定時は `#1B6B4A` |
| CHILD-R003 | 論理削除 | 「非表示」は `is_active = false` に更新。一覧画面は `is_active = true` のみ表示 |
| CHILD-R004 | ソート順 | `name_kana ASC`（ふりがな昇順） |
| CHILD-R005 | 支援領域タグ | 健康・生活 / 運動・感覚 / 認知・行動 / 言語・コミュニケーション / 人間関係・社会性 の5種類 |
| CHILD-R006 | 物理削除 | 児童一覧画面のUIから削除可能。DBでは `daily_records` がCASCADE削除される |

**児童登録データ:**

| フィールド | 必須 | デフォルト | 備考 |
|---|---|---|---|
| `name` | Yes | -- | 氏名 |
| `name_kana` | No | `null` | ふりがな（ソート用） |
| `birth_date` | No | `null` | 生年月日 |
| `school` | No | `null` | 学校名 |
| `grade` | No | `null` | 学年 |
| `icon_color` | Yes | `#1B6B4A` | 8色から選択 |
| `goals` | No | `[]` | 個別目標（複数可） |
| `domain_tags` | No | `[]` | 支援領域タグ |
| `is_active` | -- | `true` | 自動設定 |

### 2.4 日次記録

#### 概要

児童一人ひとりの1日の支援記録を入力・管理するドメイン。1児童1日1記録のユニーク制約があり、記録入力後は未記録の次の児童に自動遷移する効率的なワークフローを提供する。

#### フロー図

```
[ホーム画面: 未記録の児童をタップ]
         ↓
[記録入力画面]
  ├── 新規: arrivalTime に現在時刻を自動入力
  └── 既存: 全フィールドを既存データで初期化
         ↓
[フォーム入力（全フィールド任意）]
         ↓
[必要に応じて AI 文章生成を実行]
         ↓
[保存ボタン]
  ├── 既存記録 → UPDATE
  └── 新規記録 → INSERT
         ↓
[自動遷移]
  ├── 次の未記録児童がいる → その児童の記録画面へ
  └── 全員記録済み → ホーム画面へ
```

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| REC-R001 | ユニーク制約 | `(child_id, date)` — 1児童1日1記録 |
| REC-R002 | 空文字列処理 | `memo`、`ai_text` の空文字列は `null` に変換して保存 |
| REC-R003 | 記録者自動設定 | `recorded_by` に現在ユーザーのIDを自動設定 |
| REC-R004 | 施設ID自動設定 | `facility_id` に現在ユーザーの施設IDを自動設定 |
| REC-R005 | 更新日時自動更新 | `updated_at` はDBトリガーで自動更新 |
| REC-R006 | 削除権限 | 同施設のユーザーであれば誰でも削除可能（RLSによる制限） |
| REC-R007 | 自動遷移 | 保存後、次の未記録児童がいればその児童の記録画面へ遷移。全員記録済みならホーム画面へ |

**記録データ構造:**

| フィールド | 型 | 必須 | 備考 |
|---|---|---|---|
| `child_id` | UUID | Yes | 対象児童 |
| `date` | date | Yes | 記録日 |
| `mood` | enum | No | `good` / `neutral` / `bad` |
| `activities` | text[] | No | 選択された活動（工作, 運動, 学習, 自由遊び, SST, おやつ, 外出, 音楽, 調理） |
| `phrases` | text[] | No | 選択されたフレーズ（テキスト値） |
| `memo` | text | No | 自由記述メモ |
| `ai_text` | text | No | AI生成テキスト |
| `arrival_time` | time | No | 来所時間 |
| `departure_time` | time | No | 退所時間 |
| `pickup_method` | text | No | 送迎方法（送迎車, 保護者送迎, 自力通所, その他） |
| `recorded_by` | UUID | No | 記録者ID |
| `facility_id` | UUID | Yes | 施設ID（自動設定） |

#### エッジケース

| ケース | 発生条件 | 処理 |
|--------|---------|------|
| 同日重複登録 | 同一児童に対して同日の記録を再度INSERTしようとした場合 | ユニーク制約違反。既存記録のUPDATEとして処理 |
| 全フィールド空の保存 | 全て未入力のまま保存ボタンを押した場合 | 保存可能（全フィールド任意のため）。`recorded_by` と `facility_id` のみ自動設定 |
| 記録者と閲覧者の不一致 | 他のスタッフが入力した記録を別のスタッフが閲覧・編集 | 同施設であれば閲覧・編集可能（RLSは施設単位で制御） |

→ 関連: 03_DB設計書.md

### 2.5 AI支援記録生成

#### 概要

Claude APIを使用して、スタッフが入力した児童の活動情報やフレーズから、専門的な支援記録文を自動生成するドメイン。APIエンドポイント `POST /api/generate-record` で処理される。

#### フロー図

```
[記録入力画面: AI生成ボタン押下]
         ↓
[入力データ収集（児童情報 + 気分 + 活動 + フレーズ + メモ）]
         ↓
[POST /api/generate-record]
         ↓
[気分の日本語変換 + 年齢計算]
         ↓
[システムプロンプト + ユーザープロンプト構成]
         ↓
[Claude API 呼び出し]
         ↓
[生成テキストを ai_text フィールドに反映]
```

#### プロンプト設計

AI記録生成のモデル設定とプロンプト構造を以下に定義する。

| 項目 | 値 |
|------|---|
| モデル | `claude-sonnet-4-5-20250929` |
| max_tokens | 500 |

**システムプロンプトの要約:**
- 役割: 放課後等デイサービスの支援員
- 出力条件: 100~200文字の支援記録文
- トーン: 客観的かつ温かみのある専門的な記録
- 制約: 事実に基づく記述、推測は含めない

**ユーザープロンプトに含まれる情報:**
- 児童情報（名前、年齢、学校、学年）
- 本日の様子（気分: 良好 / 普通 / やや不調）
- 活動内容
- 観察された行動（選択されたフレーズ）
- スタッフメモ

**気分の日本語変換:**

| キー | 日本語 |
|------|--------|
| `good` | 良好 |
| `neutral` | 普通 |
| `bad` | やや不調 |

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| AI-R001 | エラーハンドリング | AI APIエラー時はHTTP 500 + エラーメッセージを返す。クライアント側は `alert` でエラー表示 |
| AI-R002 | 年齢計算 | `birth_date` から現在の年齢を算出し、プロンプトに含める |
| AI-R003 | 出力形式 | レスポンスは `{ "text": "生成されたテキスト" }` |

#### エッジケース

| ケース | 発生条件 | 処理 |
|--------|---------|------|
| APIタイムアウト | Anthropic APIが応答しない場合 | HTTP 500エラー。ユーザーにはalertで通知 |
| 入力データが極端に少ない | 気分・活動・フレーズ・メモが全て未入力 | APIは呼び出し可能（児童情報のみでもプロンプト構成可能） |
| APIキー未設定 | `ANTHROPIC_API_KEY` が環境変数にない | サーバーエラー。開発時に即座に気付ける |

→ 関連: 06_API設計書.md、07_infrastructure.md §6.3（Anthropic）

### 2.6 フレーズバンク

#### 概要

記録入力時に選択できる定型フレーズを管理するドメイン。全施設共通のデフォルトフレーズ（20件）と、施設ごとのカスタムフレーズの2種類がある。

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| PHR-R001 | フレーズ種別 | デフォルト（`facility_id = null`, `is_default = true`）とカスタム（`facility_id = 施設ID`, `is_default = false`）の2種類 |
| PHR-R002 | デフォルト保護 | デフォルトフレーズは編集・削除不可 |
| PHR-R003 | 表示優先ルール | 児童の `domain_tags` に一致するカテゴリのフレーズを優先表示 |
| PHR-R004 | ソート | カテゴリ内は `sort_order ASC` でソート |
| PHR-R005 | RLS | SELECT時はデフォルト + 自施設のフレーズを表示。DELETE時は `is_default = false` かつ自施設のみ |

**デフォルトフレーズ（カテゴリ別件数）:**

| カテゴリ | 件数 |
|---------|------|
| 健康・生活 | 5件 |
| 運動・感覚 | 3件 |
| 認知・行動 | 4件 |
| 言語・コミュニケーション | 4件 |
| 人間関係・社会性 | 4件 |
| **合計** | **20件** |

### 2.7 帳票出力

#### 概要

日次記録データをもとに、業務日誌、サービス提供記録、月間出席サマリーの3種類の帳票を画面表示・印刷するドメイン。

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| RPT-R001 | 業務日誌 | 1日の全児童の記録を一覧化。A4縦。ヘッダーに施設名・日付・出席者数、フッターに管理者署名欄 |
| RPT-R002 | サービス提供記録 | 個別児童の1日の支援記録。A4縦。利用者情報セクション + 支援記録セクション |
| RPT-R003 | 月間出席サマリー | 月単位の児童出席状況マトリクス。A4横。出席は「◯」、欠席は空白 |
| RPT-R004 | 出席判定 | `daily_records` の存在で出席を判定。`Set<"childId_date">` でO(1)ルックアップ |
| RPT-R005 | 月間集計 | 最終列に児童別月間出席日数、最終行に日別出席者数合計、右下に月間総出席数 |

**業務日誌の表示項目:**

| 列 | データ元 |
|---|---|
| 児童名 | `children.name` |
| 来所 | `daily_records.arrival_time` |
| 退所 | `daily_records.departure_time` |
| 活動内容 | `daily_records.activities`（カンマ区切り） |
| 支援記録まとめ | `daily_records.ai_text` |
| 送迎 | `daily_records.pickup_method` |
| 記録者 | `profiles.display_name`（`recorded_by` で紐付け） |

### 2.8 管理画面ロジック

#### 概要

スーパー管理者向けの管理画面（`apps/admin`）のビジネスロジック。全施設のデータを横断的に管理する。APIは `/api/admin-data` に統一し、`type` クエリパラメータでアクションを切り替える。

#### 重要ルール

| ルールID | ルール | 詳細 |
|---------|-------|------|
| ADM-R001 | API統一 | 全エンドポイントは `/api/admin-data` で統一。`type` パラメータでアクション切替 |
| ADM-R002 | 認証チェック | 全APIリクエストで Cookie `admin_session` の存在をチェック。なければHTTP 401 |
| ADM-R003 | RLSバイパス | Supabase `service_role` keyを使用（全施設データにアクセス） |
| ADM-R004 | 削除未実装 | 施設削除・ユーザー削除は管理画面から不可 |
| ADM-R005 | 即時実行 | ロール変更・スーパー管理者権限変更は確認ダイアログなしで即座に実行される |
| ADM-R006 | 施設無効化の制限 | `facilities.is_active = false` にしても、所属ユーザーのログインやデータアクセスはRLSレベルでは制限されない（UI表示制御のみ） |

**GETリクエスト一覧:**

| type | 説明 | 戻り値 |
|---|---|---|
| `stats` | ダッシュボード統計 | 施設数、ユーザー数、記録数カウント + 最近の記録10件 |
| `facilities` | 施設一覧 | 全施設 + スタッフ数 |
| `facility` | 施設詳細 | 施設データ + スタッフ数 + 記録数 |
| `facility-records` | 施設の記録 | `daily_records`（50件上限、日付フィルター） |
| `facility-staff` | 施設のスタッフ | `profiles` 一覧（`created_at DESC`） |
| `facility-children` | 施設の児童 | `children` 一覧（`name ASC`） |
| `users` | ユーザー一覧 | `profiles` + `facilities`（結合、`created_at DESC`） |
| `user` | ユーザー詳細 | profile + facility（結合） |
| `reports` | レポートデータ | 日付範囲内の記録件数 + 施設別集計 |

**POSTリクエスト一覧:**

| type | 説明 | リクエストボディ |
|---|---|---|
| `create-facility` | 施設作成 | `{ name, plan?, notes? }` |
| `update-facility` | 施設更新 | `{ id, name, plan, notes?, is_active }` |
| `update-role` | ロール変更 | `{ userId, role }` |
| `update-super-admin` | スーパー管理者変更 | `{ userId, is_super_admin }` |

**データ取得制約:**

| データ | 取得上限 | ソート |
|---|---|---|
| ダッシュボード最近の記録 | 10件 | `created_at DESC` |
| 施設の記録 | 50件 | `created_at DESC` |
| 施設一覧 | 上限なし | -- |
| ユーザー一覧 | 上限なし | `created_at DESC` |

**注意:** ページネーションは未実装。データ量の増加に伴い、全件取得がパフォーマンスに影響する可能性がある。

## 3. データ整合性ルール

### 3.1 ユニーク制約

ビジネスルールとして一意性が求められるデータの一覧を以下に定義する。

| テーブル | カラム / 組み合わせ | 制約 | 違反時の処理 |
|---------|-------------------|------|-------------|
| `daily_records` | (`child_id`, `date`) | UNIQUE複合 | 既存記録をUPDATE（UPSERT） |
| `attendances` | (`child_id`, `date`) | UNIQUE複合 | 1児童1日1出席 |
| `profiles` | `id` | UNIQUE | `auth.users.id` と1:1 |

→ 関連: 03_DB設計書.md

### 3.2 カスケード削除・更新

親エンティティの削除時における関連データの連鎖動作を以下に定義する。

| 親テーブル | 子テーブル | 削除時 | 理由 |
|-----------|---------|--------|------|
| `auth.users` | `profiles` | CASCADE | ユーザー退会時はプロフィールも削除 |
| `children` | `daily_records` | CASCADE | 児童削除時は全記録も削除 |
| `children` | `attendances` | CASCADE | 児童削除時は出席データも削除 |

→ 関連: 03_DB設計書.md

| トリガー/処理名 | 対象テーブル | タイミング | 処理内容 |

| `update_modified_at` | `children` | BEFORE UPDATE | `updated_at` を現在時刻に自動更新 |
| `update_modified_at` | `daily_records` | BEFORE UPDATE | `updated_at` を現在時刻に自動更新 |

## 4. エラーハンドリング

### 4.1 ビジネスエラー一覧

ビジネスロジックに起因するエラーの種別と処理方針を以下に定義する。

| エラー種別 | 発生条件 | 処理 | ユーザーへの表示 |
|-----------|---------|------|----------------|
| ユニーク制約違反 | 同一児童に同日の記録を重複INSERT | 既存記録のUPDATEに切り替え | なし（自動処理） |
| 認証エラー（メインアプリ） | メール/パスワードが不正 | ログイン画面に留まる | 「メールアドレスまたはパスワードが正しくありません」 |
| 認証エラー（管理画面） | パスワードが不正 | ログイン画面に留まる | 「パスワードが正しくありません」 |
| 未認証アクセス | セッショントークンなしで保護ページにアクセス | `/login` へリダイレクト | なし（自動リダイレクト） |
| AI APIエラー | Anthropic APIからのエラーレスポンスまたはタイムアウト | HTTP 500を返却 | `alert` でエラーメッセージ表示 |
| 管理画面API未認証 | `admin_session` Cookieなしで `/api/admin-data` にアクセス | HTTP 401 `"Unauthorized"` | ログイン画面にリダイレクト |
| 権限エラー（UI） | `staff` が管理者ページにアクセス | `/settings` へリダイレクト | なし（自動リダイレクト） |

### 4.2 バリデーションルール

入力値のバリデーションルールを以下に定義する。

| 対象フィールド | ルール | フロントエンド | バックエンド | エラーメッセージ |

| メールアドレス | 有効なメール形式 | リアルタイム | Supabase Auth | 「有効なメールアドレスを入力してください」 |
| パスワード | 6文字以上 | リアルタイム | Supabase Auth | 「パスワードは6文字以上で入力してください」 |
| 表示名 | 必須 | 送信時 | アプリ層 | 「表示名を入力してください」 |
| 施設名 | 必須（新規登録時） | 送信時 | アプリ層 | 「施設名を入力してください」 |
| 児童氏名 | 必須 | 送信時 | DB制約 | 「名前を入力してください」 |
| フレーズテキスト | 必須 | 送信時 | DB制約 | 「フレーズを入力してください」 |

## 5. ナビゲーション・画面遷移ルール

### 5.1 認証ガード

認証状態に基づく画面遷移ルールを以下に定義する。

| 条件 | 動作 |

### 5.2 ロールベースアクセス制御（UIレベル）

UIレベルでのロール別アクセス制御。

| ページ | `admin` | `staff` |
|---|---|---|
| `/settings/facility` | アクセス可 | → `/settings` へリダイレクト |
| `/settings/staff` | アクセス可 | → `/settings` へリダイレクト |
| `/settings/staff/invite` | アクセス可 | → `/settings` へリダイレクト |

### 5.3 記録入力の自動遷移

記録保存後の自動遷移ロジック:

```
記録保存後:
  1. allChildren から今日の allRecords に存在しない児童を抽出
  2. 次の未記録児童が存在する場合:
     → /record/[nextChildId] へ遷移
  3. 全員記録済みの場合:
     → / （ホーム）へ遷移
```
