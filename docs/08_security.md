## 1. セキュリティ概要

本アプリケーションは児童の個人情報（氏名、生年月日、支援記録等）を扱うため、適切なセキュリティ対策が必須である。Supabase Authの認証基盤とPostgreSQLのRLS（Row Level Security）を活用し、多層的なセキュリティを実現する。

### 1.1 データ分類

システムが扱うデータの機密度レベルを以下のとおり定義する。

| 分類 | 説明 | 例 | 保護レベル |
|------|------|---|-----------|
| 機密 | 漏洩時にユーザーに直接的な被害が生じるデータ | パスワードハッシュ、`SERVICE_ROLE_KEY`、`ANTHROPIC_API_KEY`、`ADMIN_PASSWORD`、児童の個人情報（氏名、生年月日） | 暗号化必須・アクセス制御・ログ記録 |
| 内部 | 漏洩時にサービス運営に支障をきたすデータ | ユーザーメールアドレス、表示名、支援記録、活動内容、気分記録 | アクセス制御（RLS）・ログ記録 |
| 公開 | 漏洩しても問題のないデータ | デフォルトフレーズ、活動種別マスタ、アプリ設定 | 改ざん防止のみ |

### 1.2 リスク評価

主要なセキュリティリスクとその対策状況を以下にまとめる。

| リスク | 影響度 | 対策状況 | 対策の詳細セクション |
|--------|--------|---------|-------------------|
| 認証情報の漏洩 | 高 | 実装済み | §2.1 |
| テナント間データ漏洩 | 高 | 実装済み（RLS） | §4.2 |
| SQLインジェクション | 高 | 実装済み（Supabase SDK） | §3.3 |
| 児童個人情報の不正アクセス | 高 | 実装済み（RLS + RBAC） | §3, §4 |
| XSS攻撃 | 中 | 実装済み（React DOM escape） | §6.1 |
| CSRF攻撃 | 中 | 実装済み（SameSite Cookie） | §6.2 |
| セッションハイジャック | 中 | 実装済み（HttpOnly Cookie） | §2.3 |
| 依存パッケージの脆弱性 | 中 | 運用対応 | §7 |
| Service Role Key漏洩 | 高 | 実装済み（サーバーサイド限定） | §6.3 |

→ 関連: 01_要件定義書.md

## 2. 認証セキュリティ

### 2.1 認証方式

本システムではメインアプリとAdmin管理画面で異なる認証方式を採用する。

**メインアプリ（`apps/main`）:**

| 項目 | 値 |
|------|---|
| 認証方式 | Supabase Auth（メール・パスワード） |
| セッション管理 | JWT（HttpOnly Cookie、Supabaseが自動管理） |
| トークン有効期限 | アクセストークン 1時間 / リフレッシュトークン 30日 |
| パスワードポリシー | 最低6文字 |
| アカウントロック | なし（Supabaseデフォルト） |
| MFA | 未対応 |

**管理画面（`apps/admin`）:**

| 項目 | 値 |
|------|---|
| 認証方式 | 共有パスワード認証（Supabase Authとは独立） |
| セッション管理 | Cookie（`admin_session`） |
| セッション有効期限 | 24時間 |
| パスワード管理 | 環境変数 `ADMIN_PASSWORD` で設定 |

### 2.2 認証フロー

**メインアプリ ログインフロー:**

```
ユーザー          フロントエンド        Supabase Auth       PostgreSQL
  |                    |                    |                   |
  |-- ログイン要求 -->  |                    |                   |
  |                    |-- signInWithPw --> |                   |
  |                    |                    |-- bcrypt検証 -->  |
  |                    |                    |<-- 結果 --------- |
  |                    |<-- JWT(Cookie) --- |                   |
  |<-- ホーム画面 ---- |                    |                   |
```

**管理画面 ログインフロー:**

```
管理者            フロントエンド        API Route
  |                    |                    |
  |-- PW入力 -------> |                    |
  |                    |-- POST /api/login ->|
  |                    |                    |-- ADMIN_PASSWORD と比較
  |                    |<-- Cookie設定 ----- |
  |<-- ダッシュボード - |                    |
```

### 2.3 セッション管理

**メインアプリ:**

- トークン保存場所: HttpOnly Cookie（XSS対策）
- リフレッシュ戦略: Next.jsミドルウェアでリクエストごとにトークン有効性を検証し、期限切れ時は自動リフレッシュ
- ログアウト時: Supabase `signOut()` でサーバー側セッション無効化 + Cookie削除
- 複数デバイス: 各デバイスに独立したセッション

**管理画面:**

- Cookie設定: `httpOnly: true`, `secure: true`（本番）, `sameSite: "lax"`, `maxAge: 86400`（24時間）
- ログアウト時: 同Cookie名で `maxAge: 0` を設定（即時失効）
- ミドルウェアで全リクエストに対し `admin_session` Cookieの存在を検証

→ 関連: 04_技術設計書.md

## 3. 認可

### 3.1 ロール定義

システムのロール（役割）を以下のとおり定義する。

| ロール | 説明 | 権限レベル |
|--------|------|-----------|
| `admin` | 施設管理者。施設設定・スタッフ管理を含む全機能へのアクセス | 自施設リソースのフルCRUD + スタッフ管理 + 施設設定 |
| `staff` | 一般スタッフ。日常業務（記録入力、児童管理等）のみ | 自施設リソースのCRUD（施設設定・スタッフ管理を除く） |
| `super_admin` | スーパー管理者。全施設のデータを閲覧・更新可能 | 全施設リソースのRead + Update |

### 3.2 権限マトリクス（RBAC）

各ロールが操作可能なリソースと操作の一覧を以下に定義する。

| 機能 | `admin` | `staff` |
|------|---------|---------|
| 日次記録の入力・編集 | CRUD | CRUD |
| 児童の登録・編集 | CRUD | CRUD |
| 帳票の閲覧・印刷 | R | R |
| フレーズの管理 | CRUD | CRUD |
| 記録履歴の閲覧・削除 | RD | RD |
| 施設設定の変更 | RU | -- |
| スタッフの管理 | RU | -- |
| ロールの変更 | U | -- |
| 招待リンクの生成 | C | -- |
| 管理画面（Admin app）アクセス | `super_admin` のみ | -- |

### 3.3 実装レイヤー

認可は多層防御の考え方で、3つのレイヤーで実装する。

| レイヤー | 実装方法 | 対象 |
|---------|---------|------|
| DB（RLS） | Supabase Row Level Securityポリシー | データアクセス制御（主防御）。`facility_id` による施設間分離 |
| アプリケーション | ミドルウェア（Next.js Middleware） | 未認証ユーザーの `/login` へのリダイレクト |
| フロントエンド | `useProfile` フックの `isAdmin` フラグ | 管理者限定UIの表示/非表示制御（UX目的、セキュリティ境界ではない） |

`staff` ユーザーが管理者ページ（`/settings/facility`、`/settings/staff`）にアクセスした場合、`/settings` へリダイレクトする。

→ 関連: 03_DB設計書.md（RLSポリシー）

## 4. データ保護

### 4.1 データ暗号化

データの暗号化方式を以下に定義する。

| 対象 | 暗号化方式 | 備考 |
|------|-----------|------|
| パスワード | bcrypt（コストファクタ10） | Supabase Authが自動処理 |
| DBデータ（保存時） | AES-256（Supabase管理） | Supabaseプラットフォームが提供 |
| 通信 | TLS 1.2+ | Vercel / Supabase ともに自動SSL |
| ローカルストレージ | 暗号化なし | 機密データは保存しない方針 |

### 4.2 マルチテナント分離

本システムは `facility_id` による行レベル分離を採用する。

| 項目 | 方式 |
|------|------|
| テナント識別 | `facility_id` カラムで識別 |
| 分離レベル | 行レベル分離（RLS） |
| 実装方法 | 全データテーブルに `facility_id` カラム + RLSポリシー |

全データテーブルに `facility_id` カラムを設置し、Supabase の RLS ポリシーで認証ユーザーの `profiles.facility_id` と一致する条件を全操作に適用する。ユーザーは自施設のデータのみアクセス可能。

各テーブルのRLSポリシー概要:

| テーブル | SELECT | INSERT | UPDATE | DELETE |
|---|---|---|---|---|
| `facilities` | 自施設のみ | 認証済みユーザー | `admin` のみ | -- |
| `profiles` | 同施設のみ | 自分自身（`id = auth.uid()`） | 自分 or `admin`（同施設） | -- |
| `children` | 同施設のみ | 同施設 | 同施設 | 同施設 |
| `daily_records` | 同施設のみ | 同施設 | 同施設 | 同施設 |
| `attendances` | 同施設のみ | 同施設 | 同施設 | -- |
| `phrase_bank` | デフォルト + 同施設 | 同施設 | 同施設 | `is_default = false` かつ同施設 |

`super_admin`（`profiles.is_super_admin = true`）は専用RLSポリシーにより全施設のSELECT/UPDATEが許可される。

→ 関連: 03_DB設計書.md

### 4.3 データ分類別保護方針

§1.1 で定義したデータ分類ごとの具体的な保護策を以下に記載する。

| 分類 | 保存場所 | アクセス制御 | バックアップ |
|------|---------|------------|------------|
| 機密 | Supabase DB（RLS適用） | `admin` + 本人のみ / サーバーサイドのみ | 日次自動バックアップ（Proプラン） |
| 内部 | Supabase DB（RLS適用） | RLSで同施設のみ | 日次自動バックアップ（Proプラン） |
| 公開 | Supabase DB | 全ユーザーRead可（デフォルトフレーズ等） | 日次自動バックアップ（Proプラン） |

### 4.4 個人情報保護

本システムが扱う個人情報の一覧と取り扱い方針を以下に定義する。

| 個人情報 | 収集目的 | 保存期間 | 削除方針 |
|---------|---------|---------|---------|
| メールアドレス | アカウント認証・ログイン | アカウント存続中 | 退会時に削除 |
| 表示名 | アプリ内での識別表示 | アカウント存続中 | 退会時に削除 |
| 児童氏名 | 支援記録の対象特定 | 施設利用中 | 論理削除（`is_active = false`） |
| 児童生年月日 | 年齢計算（AI記録生成で使用） | 施設利用中 | 論理削除 |
| 児童学校・学年 | 支援記録の補助情報 | 施設利用中 | 論理削除 |
| 支援記録 | 日々の支援内容の記録・帳票出力 | 無期限 | 手動削除可能 |

**個人情報保護対策:**

- データベースの保管時暗号化（Supabase AES-256）
- HTTPS による通信暗号化（TLS 1.2/1.3）
- 児童の物理削除禁止（論理削除: `is_active = false`）
- RLSによる施設間データ分離
- アクセスログの記録（Supabaseダッシュボード）

→ 関連: 07_infrastructure.md §9.2（機密データ除外ルール）

## 5. 通信セキュリティ

### 5.1 HTTPS

- 全通信をHTTPS経由で行う
- Vercel / Supabase ともに自動SSL証明書（Let's Encrypt）
- HTTP → HTTPSの自動リダイレクト

### 5.2 セキュリティヘッダー

Vercelが自動設定するセキュリティヘッダーを以下に記載する。

| ヘッダー | 値 | 目的 |
|---------|---|------|
| `X-Content-Type-Options` | `nosniff` | MIMEスニッフィング防止 |
| `X-Frame-Options` | `SAMEORIGIN` | クリックジャッキング防止 |
| `Strict-Transport-Security` | `max-age=63072000` | HTTPS強制 |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | リファラー漏洩防止 |

### 5.3 CORS設定

| 項目 | 設定値 |
|------|--------|
| 許可オリジン | Supabaseダッシュボードで許可オリジンを設定 |
| 許可メソッド | `GET, POST, PUT, DELETE` |
| 許可ヘッダー | `Authorization, Content-Type` |
| クレデンシャル | `true` |

### 5.4 APIセキュリティ

API通信におけるセキュリティ対策。

| 対策 | 内容 |
|------|------|
| 認証 | Bearer Token（JWT） |
| Rate Limiting | Supabaseのデフォルトレート制限 |
| APIキー | `anon` キー（公開用）のみクライアントで使用 |
| Service Role Key | サーバーサイドのみ（管理画面APIルート）。クライアントに露出しない |

→ 関連: 06_API設計書.md

## 6. フロントエンドセキュリティ

### 6.1 XSS（クロスサイトスクリプティング）対策

フロントエンドにおけるXSS対策を以下のとおり実施する。

| 対策 | 実装方法 | 適用範囲 |
|------|---------|---------|
| DOMエスケープ | Reactの自動エスケープ機構 | 全コンポーネント（デフォルト） |
| `dangerouslySetInnerHTML` 禁止 | コードレビューで検出 | プロジェクト全体 |
| ユーザー入力の安全な表示 | ユーザー入力値を直接DOMに挿入しない | 全入力フィールド |

### 6.2 CSRF（クロスサイトリクエストフォージェリ）対策

CSRF対策の実装状況。

| 対策 | 実装方法 | 備考 |
|------|---------|------|
| SameSite Cookie | `SameSite=Lax`（Supabaseデフォルト） | トークンの自動送信を制限 |
| JWT認証 | Supabase AuthのJWTトークン | Cookie以外にAuthorizationヘッダーで送信 |
| 管理画面Cookie | `sameSite: "lax"` | 管理画面のセッションCookie |

### 6.3 機密情報のクライアント側管理

クライアントサイドにおける機密情報の管理方針を以下に定義する。

| 変数 | 公開範囲 | 機密性 |
|------|---------|--------|
| `NEXT_PUBLIC_SUPABASE_URL` | クライアント + サーバー | 低（公開可） |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | クライアント + サーバー | 低（RLSで保護） |
| `SUPABASE_SERVICE_ROLE_KEY` | サーバーのみ（Admin） | 高（非公開必須） |
| `ANTHROPIC_API_KEY` | サーバーのみ（メインアプリ） | 高（非公開必須） |
| `ADMIN_PASSWORD` | サーバーのみ（Admin） | 高（非公開必須） |

追加のルール:

- `NEXT_PUBLIC_` プレフィックスの変数に機密情報を含めない
- ローカルストレージに認証トークンやパスワードを保存しない
- `console.log` で機密データを出力しない
- 本番環境ではソースマップを公開しない

## 7. 依存関係の脆弱性管理

### 7.1 npm auditポリシー

脆弱性の検出と対応の方針を以下のとおり定義する。

| 重大度 | 対応期限 | 対応方法 |
|--------|---------|---------|
| critical | 24時間以内 | 即時パッチ適用。パッチがない場合は代替パッケージへ移行 |
| high | 1週間以内 | パッチ適用またはワークアラウンド実施 |
| moderate | 次回リリースまで | 定期メンテナンスで対応 |
| low | 任意 | リスク受容可。次回大規模更新時に対応 |

### 7.2 定期チェック

- `npm audit` を週1回手動で実行
- `npm audit --audit-level=high` で high 以上を検出時は優先対応
- 結果はチーム内で共有

### 7.3 Dependabot / Renovate 設定

| 項目 | 設定値 |
|------|--------|
| ツール | 未導入 |
| 理由 | 小規模チームのため手動チェックで対応中 |
| 導入検討時期 | チーム拡大時またはCI/CD整備時 |

→ 関連: 07_infrastructure.md §7

## 8. インシデント対応

### 8.1 インシデント対応フロー

```
1. 検知 ──→ 2. 初期評価 ──→ 3. 封じ込め ──→ 4. 調査 ──→ 5. 復旧 ──→ 6. 振り返り
```

| ステップ | 内容 | 目標時間 |
|---------|------|---------|
| 1. 検知 | ログ監視・ユーザー報告による検知 | -- |
| 2. 初期評価 | 影響範囲・重大度の判断 | 30分以内 |
| 3. 封じ込め | 被害拡大の防止（アカウント停止、機能無効化等） | 1時間以内 |
| 4. 調査 | 原因の特定、影響範囲の確定 | 24時間以内 |
| 5. 復旧 | サービスの正常化、脆弱性の修正 | 調査完了後 |
| 6. 振り返り | ポストモーテム実施、再発防止策の策定 | 1週間以内 |

### 8.2 重大度定義

インシデントの重大度レベルを以下に定義する。

| 重大度 | 定義 | 対応レベル |
|--------|------|-----------|
| P1（緊急） | 児童個人情報の漏洩・サービス全停止・不正アクセス | 即時対応。全作業を中断 |
| P2（高） | 一部機能停止・脆弱性の悪用可能性 | 24時間以内に対応 |
| P3（中） | セキュリティ警告・潜在的リスク | 次回リリースで対応 |

### 8.3 ログ・監査

セキュリティ監査に必要なログの記録方針を以下に定義する。

| ログ種別 | 記録内容 | 保存場所 | 保存期間 |
|---------|---------|---------|---------|
| 認証ログ | ログイン/ログアウト/失敗 | Supabase Auth Logs | 7日間（Free） |
| APIリクエストログ | メソッド、パス、ステータス | Supabase API Logs | 7日間（Free） |
| データベースログ | クエリ実行ログ | Supabase DB Logs | 7日間（Free） |
| アプリケーションログ | エラー、例外 | Vercel Functions Logs | 1時間（Free）/ 3日（Pro） |

→ 関連: 07_infrastructure.md §9

## 改善提案

| # | 提案 | 優先度 | 理由 |
|---|------|--------|------|
| 1 | 管理画面の認証をSupabase Authベースに移行 | 高 | 現在の共有パスワード認証は個人識別ができず、監査ログに記録者情報が残らない |
| 2 | `ADMIN_PASSWORD` のフォールバック値（`"Admin"`）の削除 | 高 | 環境変数未設定時にデフォルトパスワードでログイン可能になるリスクがある |
| 3 | Content-Security-Policy（CSP）ヘッダーの追加 | 中 | XSS対策をさらに強化できる |
| 4 | ロール変更・スーパー管理者権限変更時の確認ダイアログ追加 | 中 | 現在は確認なしで即座に実行される。誤操作防止のため必要 |
| 5 | Dependabotの導入 | 低 | 依存関係の脆弱性を自動検出し、PRを自動作成できる |
| 6 | 認証ログの長期保存（Proプランまたは外部ログサービス） | 中 | Freeプランでは7日間のみ。インシデント調査に不十分な場合がある |
