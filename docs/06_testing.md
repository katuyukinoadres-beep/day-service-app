## 1. テスト方針

### 1.1 テスト戦略

プロジェクト全体のテスト構成を以下に定義する。

| レベル | ツール | 対象 | 方針 |
|--------|--------|------|------|
| ビルドテスト | `next build` | TypeScript 型チェック・ビルド成功 | 全 push で CI 実行。型エラーはマージ不可 |
| リントテスト | ESLint 9 | コード品質・一貫性 | 全 push で CI 実行 |
| E2Eテスト（手動） | ブラウザ | 主要ユーザーフロー | P1 シナリオを中心にリリース前実施 |
| 手動テスト | ブラウザ | UI/UX、印刷、レスポンシブ、実機確認 | リリース前チェックリスト |

→ 関連: 08_インフラ構成書.md

### 1.2 テスト優先度

テストケースの優先度を以下の3段階で管理する。

| 優先度 | 定義 | 対応方針 |
|--------|------|---------|
| P1: 必須 | 未通過ならリリース不可。認証、データCRUD、帳票出力、ロール制御、セキュリティ | 全件リリース前に必ず実施 |
| P2: 重要 | 主要機能の異常系・境界値。画面遷移、バリデーション、フィルタ、AI機能 | リリース前に可能な限り実施 |
| P3: 推奨 | エッジケース、UI表示の細部、パフォーマンス。時間があれば実施 | 余裕がある場合に実施 |

### 1.3 テスト対象アプリ

テスト対象の2つのアプリケーションを示す。

| アプリ | URL | 説明 |
|--------|-----|------|
| メインアプリ（`apps/main`） | `http://localhost:3000` | スタッフ向け記録アプリ |
| 管理アプリ（`apps/admin`） | `http://localhost:3001` | システム管理者向けダッシュボード |

### 1.4 カバレッジ目標

| 対象 | 目標 | 測定方法 |
|------|------|---------|
| ビルドテスト | 全アプリで型エラー 0 | `next build` の成功 |
| リントテスト | ESLint エラー 0 | `npm run lint:main` / `npm run lint:admin` |
| E2Eテスト（手動） | P1 シナリオ 100% 通過 | テストケースの実施記録 |

### 1.5 実行方法

```bash
# メインアプリ ビルドテスト
npm run build:main

# 管理アプリ ビルドテスト
npm run build:admin

# メインアプリ Lint
npm run lint:main

# 管理アプリ Lint
npm run lint:admin

# 開発サーバー起動（手動テスト用）
npm run dev:main          # port 3000
npm run dev:admin         # port 3001

# 本番デプロイ
npx vercel --prod --yes
```

### 1.6 テスト実行環境

各テストレベルの実行場所とタイミングを定義する。

| テストレベル | 実行場所 | タイミング | 備考 |
|------------|---------|-----------|------|
| ビルドテスト | ローカル + Vercel | push / デプロイ時 | Vercel ビルドで失敗ならデプロイ中止 |
| リントテスト | ローカル | push 前 | コミット前に実行推奨 |
| E2Eテスト（手動） | ローカル / 本番 | リリース前 | 下記テストケース表に従い実施 |
| 手動テスト | 本番環境 | デプロイ後 | 実機 + DevTools で確認 |

## 2. テストデータ

### 2.1 テスト用アカウント

テスト環境で使用するアカウントを定義する。

| メールアドレス | パスワード | ロール | 用途 |
|--------------|-----------|--------|------|
| `test-admin@example.com` | `Test1234` | admin | 管理者機能のテスト（施設設定、スタッフ管理） |
| `test-staff@example.com` | `Test1234` | staff | スタッフ機能のテスト（記録入力、閲覧） |
| `test-empty@example.com` | `Test1234` | admin | データなし状態のテスト（児童未登録） |

管理アプリ用のパスワードは環境変数 `ADMIN_PASSWORD` で設定する。

### 2.2 データリセット手順

テスト実行前にデータを初期状態に戻すための手順。

1. テスト用 Supabase プロジェクトで SQL エディタを開く
2. テストデータ削除クエリを実行（`daily_records` → `children` → `profiles` → `facilities` の順）
3. シードデータを投入（テスト用アカウント・施設・児童を再作成）

→ 関連: 03_DB設計書.md

## 改善メモ

- Vitest によるユニットテスト・コンポーネントテストの導入を検討し、テスト戦略に追加する
- Playwright による E2E テスト自動化を導入し、P1 シナリオの CI 自動実行を実現する
- API 統合テスト（Supabase RPC/REST に対する自動テスト）のセクションを追加する
- テストデータのシードスクリプト（`npm run db:reset:test`）を実装し、コマンドを更新する
- 管理アプリ UI テスト（`AUI-001` 〜 `AUI-003`）を追加し、サイドバー・モバイルメニューの動作を検証する
- 既存テスト仕様書との差分として、管理アプリ UI テスト3件が本テンプレートでは統合されていないため、必要に応じて追加する
