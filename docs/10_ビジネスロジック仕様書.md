# ビジネスロジック仕様書

## 1. 概要

本書は「ぱっと記録」のビジネスルール、ワークフロー、データ処理ロジックを定義する。コードを再実装する際に、単純なCRUD操作では表現しきれない業務知識を保全することが目的である。

---

## 2. 認証・ユーザー管理

### 2.1 新規登録フロー（施設オーナー）

```
1. メール + パスワード + 表示名 + 施設名 を入力
2. Supabase Auth でアカウント作成（signUp）
3. 自動サインイン（signInWithPassword）
4. facilities テーブルに施設を INSERT
5. profiles テーブルにプロフィールを INSERT（role = "admin"）
6. ホーム画面へリダイレクト
```

**重要ルール:**
- 新規登録時は必ず施設を作成する（施設なしのユーザーは存在しない）
- 最初のユーザーは必ず admin ロール

### 2.2 スタッフ招待フロー

```
1. admin が /settings/staff/invite で招待リンクを生成
2. リンク形式: {origin}/join?facility={facilityId}
3. 招待されたスタッフがリンクを開く
4. メール + パスワード + 表示名 を入力
5. アカウント作成 → サインイン → プロフィール作成（role = "staff"）
```

**重要ルール:**
- 招待リンク経由のユーザーは必ず `role = "staff"`
- facility_id はURLパラメータから取得（施設選択UIなし）

### 2.3 ロール変更ルール

| 操作 | 実行者 | 対象 |
|---|---|---|
| staff → admin | admin | 同施設の他スタッフ |
| admin → staff | admin | 同施設の他スタッフ |
| 自分のロール変更 | — | 不可（UIで無効化） |

### 2.4 管理画面認証（admin アプリ）

- パスワード認証（環境変数 `ADMIN_PASSWORD` と比較）
- Cookie `admin_session = "authenticated"`（httpOnly、24時間有効）
- Supabase Auth とは独立した認証機構
- API ルートは service_role key を使用（RLS バイパス）

## 3. マルチテナント分離

### 3.1 データ分離方式

- 全テーブルに `facility_id` カラム
- RLS（Row Level Security）で施設単位のデータ分離
- ユーザーの `facility_id` は `profiles` テーブルから取得

### 3.2 RLS ポリシー概要

| テーブル | SELECT | INSERT | UPDATE | DELETE |
|---|---|---|---|---|
| facilities | 自施設のみ | 認証済みユーザー | admin のみ | — |
| profiles | 同施設のみ | 自分自身（id=auth.uid()） | 自分 or admin（同施設） | — |
| children | 同施設のみ | 同施設 | 同施設 | 同施設 |
| daily_records | 同施設のみ | 同施設 | 同施設 | 同施設 |
| attendances | 同施設のみ | 同施設 | 同施設 | — |
| phrase_bank | デフォルト + 同施設 | 同施設 | 同施設 | is_default=false かつ同施設 |

### 3.3 スーパー管理者

- `profiles.is_super_admin = true` のユーザーは全施設のデータを閲覧・更新可能
- 専用 RLS ポリシーで全テーブルの SELECT/UPDATE を許可

## 4. 児童管理

### 4.1 児童登録データ

| フィールド | 必須 | デフォルト | 備考 |
|---|---|---|---|
| name | YES | — | 氏名 |
| name_kana | NO | null | ふりがな（ソート用） |
| birth_date | NO | null | 生年月日 |
| school | NO | null | 学校名 |
| grade | NO | null | 学年 |
| icon_color | YES | `#1B6B4A` | 8色から選択 |
| goals | NO | `[]` | 個別目標（複数可） |
| domain_tags | NO | `[]` | 支援領域タグ |
| is_active | — | `true` | 自動設定 |

### 4.2 支援領域タグ（DOMAIN_TAGS）

```
1. 健康・生活
2. 運動・感覚
3. 認知・行動
4. 言語・コミュニケーション
5. 人間関係・社会性
```

### 4.3 論理削除

- 児童の「非表示」は `is_active = false` に更新（物理削除は児童一覧画面のUIから可能だが、DB上は CASCADE で削除される）
- 一覧画面は `is_active = true` のみ表示
- ソート順: `name_kana ASC`

## 5. 日次記録

### 5.1 記録データ構造

| フィールド | 型 | 必須 | 備考 |
|---|---|---|---|
| child_id | UUID | YES | 対象児童 |
| date | date | YES | 記録日 |
| mood | enum | NO | good / neutral / bad |
| activities | text[] | NO | 選択された活動 |
| phrases | text[] | NO | 選択されたフレーズ（テキスト値） |
| memo | text | NO | 自由記述メモ |
| ai_text | text | NO | AI生成テキスト |
| arrival_time | time | NO | 来所時間 |
| departure_time | time | NO | 退所時間 |
| pickup_method | text | NO | 送迎方法 |
| recorded_by | UUID | NO | 記録者ID |
| facility_id | UUID | YES | 施設ID（自動設定） |

**ユニーク制約:** `(child_id, date)` — 1児童1日1記録

### 5.2 活動種別（ACTIVITIES）

```
工作, 運動, 学習, 自由遊び, SST, おやつ, 外出, 音楽, 調理
```

### 5.3 気分（MOODS）

| キー | 絵文字 | ラベル |
|---|---|---|
| good | 😊 | 良好 |
| neutral | 😐 | 普通 |
| bad | 😢 | 不調 |

### 5.4 送迎方法（PICKUP_METHODS）

```
送迎車, 保護者送迎, 自力通所, その他
```

### 5.5 記録入力ワークフロー

```
1. ホーム画面で未記録の児童をタップ
2. 記録入力画面が開く
   - 新規: arrivalTime に現在時刻を自動入力
   - 既存: 全フィールドを既存データで初期化
3. フォーム入力（全フィールド任意）
4. 必要に応じて AI 文章生成を実行
5. 保存ボタン
   - 既存記録: UPDATE
   - 新規記録: INSERT
6. 自動遷移:
   - 次の未記録児童がいる → その児童の記録画面へ
   - 全員記録済み → ホーム画面へ
```

### 5.6 保存時のデータ処理

- 空文字列の `memo`、`ai_text` は `null` に変換
- `recorded_by` に現在ユーザーの ID を設定
- `facility_id` に現在ユーザーの施設 ID を設定
- `updated_at` は DB トリガーで自動更新

### 5.7 記録削除ルール

- 履歴画面から削除可能
- 確認ダイアログを表示後に DELETE
- 同施設のユーザーであれば誰でも削除可能（RLS による制限）

## 6. AI 支援記録生成

### 6.1 API エンドポイント

`POST /api/generate-record`

### 6.2 入力データ

```json
{
  "childName": "児童名",
  "birthDate": "2015-04-01",
  "school": "○○小学校",
  "grade": "3年生",
  "mood": "good",
  "activities": ["工作", "運動"],
  "phrases": ["元気に過ごすことができた", "体を動かして楽しく遊べた"],
  "memo": "今日は特に集中していた"
}
```

### 6.3 処理ロジック

1. **気分の日本語変換:**
   - good → "良好"
   - neutral → "普通"
   - bad → "やや不調"

2. **年齢計算:** birthDate から現在の年齢を算出

3. **システムプロンプト構成:**
   - 役割: 放課後等デイサービスの支援員
   - 出力条件: 100〜200文字の支援記録文
   - トーン: 客観的かつ温かみのある専門的な記録
   - 制約: 事実に基づく記述、推測は含めない

4. **ユーザープロンプト構成:**
   - 児童情報（名前、年齢、学校、学年）
   - 本日の様子（気分）
   - 活動内容
   - 観察された行動（フレーズ）
   - スタッフメモ

### 6.4 AI モデル設定

| 設定 | 値 |
|---|---|
| モデル | `claude-sonnet-4-5-20250929` |
| max_tokens | 500 |

### 6.5 出力

```json
{
  "text": "生成されたテキスト"
}
```

### 6.6 エラー処理

- AI API エラー → HTTP 500 + エラーメッセージ
- クライアント側は alert でエラー表示

## 7. フレーズバンク

### 7.1 フレーズの種類

| 種類 | facility_id | is_default | 編集 | 削除 |
|---|---|---|---|---|
| デフォルト | null | true | 不可 | 不可 |
| カスタム | 施設ID | false | 可 | 可 |

### 7.2 フレーズ表示の優先ルール

記録入力画面でのフレーズ表示順:
1. 児童の `domain_tags` に一致するカテゴリのフレーズを優先
2. カテゴリ内は `sort_order ASC` でソート
3. デフォルトフレーズとカスタムフレーズの両方を表示

### 7.3 デフォルトフレーズ一覧

**健康・生活（5件）:**
- 元気に過ごすことができた
- 体調が優れない様子だった
- 手洗い・うがいを自分からできた
- 食事を残さず食べることができた
- 身の回りの整理整頓ができた

**運動・感覚（3件）:**
- 体を動かして楽しく遊べた
- バランス遊びに取り組めた
- 手先を使う活動に集中できた

**認知・行動（4件）:**
- 課題に集中して取り組めた
- 順番を守って活動できた
- 指示を聞いて行動できた
- 気持ちの切り替えができた

**言語・コミュニケーション（4件）:**
- 自分の気持ちを言葉で伝えられた
- 友だちと会話を楽しめた
- 「ありがとう」「ごめんね」が言えた
- スタッフに相談することができた

**人間関係・社会性（4件）:**
- 友だちと協力して活動できた
- ルールを守って遊べた
- 自分から挨拶ができた
- 困っている友だちを助けられた

## 8. 帳票出力

### 8.1 業務日誌

**目的:** 1日の業務記録を一覧化

**データソース:**
- daily_records（対象日のみ）
- children（全児童）
- profiles（記録者名の解決）
- facilities（施設名）

**表示項目:**
| 列 | データ元 |
|---|---|
| 児童名 | children.name |
| 来所 | daily_records.arrival_time |
| 退所 | daily_records.departure_time |
| 活動内容 | daily_records.activities（カンマ区切り） |
| 支援記録まとめ | daily_records.ai_text |
| 送迎 | daily_records.pickup_method |
| 記録者 | profiles.display_name（recorded_by で紐付け） |

**印刷:**
- ヘッダー: 施設名、日付、出席者数
- フッター: 管理者署名欄、児童発達支援管理責任者署名欄
- A4縦

### 8.2 サービス提供記録

**目的:** 個別児童の1日の支援記録

**データソース:**
- daily_records（対象児童 + 対象日）
- children（対象児童の詳細）
- profiles（記録者名）
- facilities（施設名）

**セクション1 — 利用者情報:**
- 氏名、生年月日、学校/学年、支援領域（domain_tags）、個別目標（goals）

**セクション2 — 支援記録:**
- 来所/退所時間、気分、活動内容、フレーズ、メモ、AI記録テキスト、送迎方法、記録者

**印刷:** A4縦

### 8.3 月間出席サマリー

**目的:** 月単位の児童出席状況を一覧化

**データソース:**
- daily_records（対象月の全記録）
- children（is_active=true）
- facilities（施設名）

**データ処理:**
1. 対象月の daily_records を取得
2. `Set<"childId_date">` を構築（O(1) ルックアップ）
3. 当該月の日数を算出
4. 児童 × 日のマトリクスを生成

**集計:**
- セル: 出席 → "◯"、欠席 → 空白
- 最終列: 児童別月間出席日数
- 最終行: 日別出席者数合計
- 右下: 月間総出席数

**印刷:** A4横

## 9. 管理画面ビジネスロジック

### 9.1 ダッシュボード統計

| 指標 | 算出方法 |
|---|---|
| 施設数 | `COUNT(*) FROM facilities` |
| ユーザー数 | `COUNT(*) FROM profiles` |
| 総記録数 | `COUNT(*) FROM daily_records` |
| 本日の記録 | `COUNT(*) FROM daily_records WHERE date = today` |
| 本日稼働施設 | `COUNT(DISTINCT facility_id) FROM daily_records WHERE date = today` |

### 9.2 施設管理

**施設プラン:** free / basic / pro

**施設の有効/無効:** `facilities.is_active` で制御（無効化しても施設データは保持）

### 9.3 ユーザー管理

- ロール変更: admin ↔ staff（API 経由で profiles テーブルを UPDATE）
- スーパー管理者フラグ: `is_super_admin` のトグル

### 9.4 レポート

**施設別記録数集計:**
1. 日付範囲（デフォルト: 過去30日）で daily_records を取得
2. facility_id ごとにグループ化してカウント
3. 件数降順でソート
4. facility_id → 施設名の解決（別途 API 呼び出し）

## 10. データ整合性ルール

### 10.1 ユニーク制約

| テーブル | 制約 |
|---|---|
| daily_records | `(child_id, date)` — 1児童1日1記録 |
| attendances | `(child_id, date)` — 1児童1日1出席 |
| profiles | `id` — auth.users.id と1:1 |

### 10.2 外部キー制約

| 参照元 | 参照先 | ON DELETE |
|---|---|---|
| profiles.id | auth.users.id | CASCADE |
| profiles.facility_id | facilities.id | — |
| children.facility_id | facilities.id | — |
| daily_records.child_id | children.id | CASCADE |
| daily_records.recorded_by | profiles.id | — |
| attendances.child_id | children.id | CASCADE |
| phrase_bank.facility_id | facilities.id | — |

### 10.3 自動更新トリガー

- `children.updated_at`: UPDATE 時に自動更新
- `daily_records.updated_at`: UPDATE 時に自動更新

## 11. ナビゲーション・画面遷移ルール

### 11.1 認証ガード

| 条件 | 動作 |
|---|---|
| 未認証 + 保護ページ | `/login` へリダイレクト |
| 認証済 + `/login` | `/` へリダイレクト |
| `/join` | 認証不要（パブリック） |
| `/api/auth/*` | 認証不要 |

### 11.2 ロールベースアクセス制御（UI レベル）

| ページ | admin | staff |
|---|---|---|
| /settings/facility | ○ | → /settings へリダイレクト |
| /settings/staff | ○ | → /settings へリダイレクト |
| /settings/staff/invite | ○ | → /settings へリダイレクト |

### 11.3 記録入力の自動遷移

```
記録保存後:
  1. allChildren から今日の allRecords に存在しない児童を抽出
  2. 次の未記録児童が存在する場合:
     → /record/[nextChildId] へ遷移
  3. 全員記録済みの場合:
     → / （ホーム）へ遷移
```
