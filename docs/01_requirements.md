## 0. 設計哲学（北極星）

> **「放デイ職員の手間を最小化し、子供と向き合う時間を最大化する」**

すべての設計判断は、この一文に照らして「YES／NO」を決める。「便利機能だけど職員の手間が増える」設計はNG。「機能は地味でも職員の負荷が確実に下がる」設計を最優先する。

### 0.1 設計原則 7か条（NiKo提唱）

| # | 原則 | NG例 | OK例 |
|---|---|---|---|
| 1 | 入力は最小タップで完結 | 5画面遷移＋20項目入力 | チェック1タップ＋音声で完結 |
| 2 | 機械が判定できることは機械が | 加算対象判断を職員に委ねる | システムが自動判定→確認だけ |
| 3 | 転記・集計はゼロ | 月末に手集計 | 入力即集計、ダッシュに即反映 |
| 4 | 通知・連絡の自動化 | 保護者LINE手打ち | テンプレ自動生成→1タップ送信 |
| 5 | エラー前提の設計 | 入力ミスを職員に責めさせる | 警告＋AIで補完候補 |
| 6 | オフライン耐性 | 通信切れたら入力不可 | ローカル保存→復帰時同期 |
| 7 | 学習コスト最小 | マニュアル30ページ前提 | 触ったら分かるUI、紙のフォームに見た目を寄せる |

### 0.2 効果試算（参考）

Phase 1+2 合計で、職員1人あたり 約 -90分／日。5人体制なら 1日 7.5時間、月150時間が「子供時間」として戻る。

## 1. 対象施設と前提条件

Phase 1.1 の実証施設（上田くん運営、株式会社ピース）の運用前提を、事前ヒアリング（2026-04-18 Q&A 10項目）で確定。以降の設計判断はこの前提条件に基づく。

### 1.1 実証施設の運用前提

| 項目 | 前提値 | 影響 |
|------|--------|------|
| 児童数 | 10〜15名／日（仮置き確定） | 同時接続数・データ量見積もりのベース |
| スタッフ数 | 4〜6名（仮置き確定） | UI設計・権限構造のベース |
| 入力端末 | **スタッフ個人スマホ（BYOD）メイン** | PWA必須、ブラウザ完結型、MDM代替の運用設計必要 |
| 通信環境 | 施設内メイン・Wi-Fi不安定、**軽オフライン耐性必要** | Service Worker＋ローカルキャッシュで入力継続を担保 |
| 予算制約 | なし | 品質・職員負荷削減効果で判断、コスト最適化は判断軸から除外 |
| 法令 | 自治体指導で **電子保存OK** 確認済み | 紙出力は必須要件から除外、月次PDF出力は任意バックアップ扱い |
| 記入タイミング | スタッフ混在（活動中・合間・終業後） | 下書き自動保存必須、同日重複警告必要 |
| 記録単位 | 児童1人×1日×1レコード | ユニーク制約で担保 |
| 活動メニュー | 将来追加変更あり | マスタ管理＋廃止フラグ運用（物理削除しない） |
| 既存データ | 児童名簿=Excel+リタリコ+紙混在 / スタッフ=Excel / 過去紙記録=画像化済 | CSV/Excelインポート + リタリコ取込 + 手動入力の3経路、画像は閲覧専用アーカイブとして児童別タブで閲覧可能に |
| 保護者連絡 | LINE中心（80%以上） | Phase 1.3 レポート配信方式はLINE連携を最有力候補に比較検討（Phase 1.3 設計時に決定） |
| スタッフIT習熟度 | LINE等は使えるが新規アプリは要サポート | オンボーディング完結型UI（2分動画ガイド＋アプリ内FAQ）、研修時間ゼロでも立上げ可能な設計 |
| 受容度 | 中立（紙→デジタル切替への積極性は低め） | 紙併用期間2週間の移行期モードを用意 |

### 1.2 NiKo側の限界（引き継いだ既知の制約）

- 放デイの法令・制度の最新詳細は網羅していない
- 上田くんの予算感は未確認（ただし今回は予算制約なしで確定）

### 1.3 Phase 2/3 以降の宿題（本Phase 1.1 では確定していない項目）

- リタリコ脱却（Phase 3）の希望タイミング

## 2. アプリ構成（3層体制）

NiKo 構想による3層アーキテクチャ。Phase 1.1 時点の実装状況を併記。

| # | アプリ | 主な利用者 | 役割 | 実装状況 |
|---|--------|-----------|------|---------|
| **A** | ユーザーアプリ | 現場スタッフ／保護者 | 日々の記録入力・閲覧、保護者への情報共有 | ✅ 実装済（スタッフ側は `apps/main`、保護者ビューは Phase 1.3） |
| **B** | 施設管理者ダッシュボード | 事業所長（上田くん級） | 経営・運営の可視化＋意思決定支援 | ❌ **未実装**。Phase 1.2 着手、`apps/facility-admin` として追加予定 |
| **C** | システムAdminアプリ | Leon／開発側・スーパー管理者 | 全施設の横断管理・マスタ管理 | ✅ 実装済（`apps/admin`） |

> ⚠️ 注意: 現行 `apps/admin` は **C（システムAdmin）** であり、B の施設管理者ダッシュボードではない。B は新規に追加する必要がある。

### 2.1 B: 施設管理者ダッシュボードの主要機能（NiKo構想）

事業所長は「入力者ではなく**判断者**」。**入力画面ではなく可視化＋意思決定支援**が中心。

1. **リアルタイム業務状況**: 出席状況、スタッフ配置、記録入力進捗
2. **経営KPIサマリ**: 月間延べ利用者、加算取得状況、稼働率
3. **児童別ビュー**: 個別支援計画更新期限、活動傾向グラフ、保護者問い合わせ履歴
4. **スタッフ管理**: シフト、記録品質、研修・資格更新期限
5. **請求準備**: 月次サービス提供実績の自動集計、エビデンス整合性チェック
6. **経営判断支援**: 過去6ヶ月トレンド、月次予測

### 3.1 ハイブリッド戦略（自作 × SaaS）

| 機能 | 方式 | 理由 |

#### 4.1.4 紙フォーム忠実な入力フォーム仕様

**項目構成と紙→アプリマッピング**:

| 紙の項目 | アプリでの実現方法 |
|---------|------------------|
| 実施日 | 自動セット（当日）、過去日付の編集は日付ピッカーから |
| 利用者氏名 | ホーム画面からの遷移で自動セット、フォーム内では表示のみ |
| サービス提供時間（来〜退所） | 2つの時刻入力欄、将来的に出退勤打刻と連動して自動入力 |
| 担当者 | ログインユーザーを自動セット、複数担当時は追加選択 |
| 記録時間 | 保存時のタイムスタンプで代替（`updated_at`） |
| 活動内容チェック | **活動マスタ（施設カスタマイズ可）から複数選択**、詳細記入欄が紐づく項目は入力欄が展開 |
| 活動中のトピックス | 独立したフリー記述欄（音声入力対応） |
| 特記事項 | 独立したフリー記述欄（音声入力対応） |

**活動マスタの設計**:
- 施設ごとにカスタマイズ可能（Q6 I対応）
- 上田くん施設の初期データ: 眼球運動 / 宿題 / 漢字トレーニング / 計算トレーニング / その他取り組み
- 各項目に「詳細記入欄あり（フリーテキスト）」のフラグを持つ。例: 漢字トレーニングは詳細欄あり、眼球運動は詳細欄なし
- マスタ変更は施設管理者のみ可能（ロール: admin）
- 廃止は `is_active = false` で論理削除（過去記録の整合性保持）

**UI原則**:
- 紙フォームのレイアウトに見た目を寄せる（縦持ち前提、上から順に同じ構造）
- チェックボックスは大きなタップ領域（最小44x44px）
- 詳細記入欄は選択した項目のみ展開（画面を圧迫しない）
- 音声入力ボタンは各フリー記述欄の右上に配置

## 5. 非機能要件

### 5.1 パフォーマンス

- 画面遷移: 1秒以内（ネットワーク遅延除く）
- データ読み込み: 2秒以内
- 同時接続ユーザー数: 施設あたり最大20名
- オフライン時の入力応答: 200ms以内（ローカルIndexedDB）

### 5.2 可用性

- 稼働率: 99.5%以上（Vercel + Supabase SLAに準拠）
- 計画メンテナンス時のダウンタイム通知
- 軽オフライン耐性: Wi-Fi断続時もサービス提供記録の入力・保存は継続可能

### 5.3 セキュリティ

- 通信は全てHTTPS
- ユーザー認証（メール/パスワード）
- 施設単位のデータ分離（マルチテナント）
- Row Level Security（RLS）によるデータアクセス制御
- セッション管理（Cookieベース、`@supabase/ssr`）
- **BYOD前提**: 端末にデータを永続化しない（IndexedDBキャッシュはログアウト時にクリア）、退職時は `profiles.is_active = false` 等でRLSから即時遮断

### 5.4 対応環境

| 環境 | 対応範囲 |
|------|---------|
| ブラウザ | Chrome, Safari, Edge（最新2バージョン） |
| デバイス | スマートフォン（**BYOD前提の主**）、タブレット、デスクトップ |
| OS | iOS 15+, Android 12+, Windows 10+, macOS 12+ |
| 画面幅 | モバイルファースト設計（縦持ち前提、紙フォームと同じ縦長レイアウト） |
| PWA | 必須（Service Worker + Web App Manifest） |

### 5.5 アクセシビリティ

- タップターゲット最小44px
- 適切なラベル付け（`aria-label`）
- 十分なコントラスト比
- セマンティックHTML使用

### 5.6 拡張性

- 活動マスタの施設別カスタマイズ
- フレーズバンクのカスタマイズ
- 帳票テンプレートの追加
- Phase 1.2 以降の B アプリ追加

### 5.7 オンボーディング

- 初回ログイン時に2分以内で完結する動画ガイド
- 研修時間ゼロでも立ち上げ可能な UX（アプリ内 FAQ、ツアー）
- マニュアル30p前提の設計は禁止

## 6. スコープ外

- 完全オフライン対応（軽オフライン耐性は必須、完全オフラインは Phase 外）
- ネイティブアプリ（Webアプリ/PWAとして提供）
- 多言語対応（日本語のみ、ユーザー基盤が日本限定のため）
- リアルタイム共同編集（同時編集の競合解決は対象外、最終更新優先）
- 決済・課金機能（現時点では無料提供、将来 SaaS 展開時に検討）

## 8. 制約事項

- インターネット接続が必須（完全オフラインは非対応。軽オフライン耐性はあり）
- 児童の物理削除は行わず、論理削除（`is_active` フラグ）で対応
- デフォルトフレーズは編集・削除不可
- **活動マスタの物理削除は行わず、論理削除**（`is_active` フラグ）で対応。過去記録の整合性保持のため
- 1施設あたりの児童数は実運用上100名程度を想定
- Supabase Free プラン: DB 500MB、MAU 50,000
- Vercel: 3プロジェクト構成予定（main + admin + 将来 facility-admin）
- Anthropic API: 従量課金（AI記録生成のコスト管理が必要）

## 9. 用語集

| 用語 | 説明 |
|------|------|
| 放課後等デイサービス | 障害のある子どもに対し、放課後や休日に生活能力向上のための支援を行うサービス |

| 活動中トピックス | 紙フォームのフリー記述欄。活動中の様子・エピソードを記録 |
| 特記事項 | 紙フォームのフリー記述欄。異常事態・重要な気づきを記録 |
| フレーズバンク | 記録入力を効率化するための定型文の集合。デフォルト（全施設共通）とカスタム（施設固有）がある |
| 支援記録まとめ | 本日の支援記録をまとめた文章。手入力またはAI（Claude Opus 4.7）で下書き生成が可能 |
| Admin管理画面 | スーパー管理者用（C層）の管理ダッシュボード。`apps/admin` として独立デプロイ |
| 施設管理者ダッシュボード | 事業所長（B層）用のダッシュボード。将来 `apps/facility-admin` として追加予定 |
| RLS | Row Level Security。PostgreSQLのデータベースレベルで行単位のアクセス制御を行う仕組み |
| マルチテナント | `facility_id` による論理分離 + RLS で、施設ごとのデータを完全に分離する方式 |
| BYOD | Bring Your Own Device。スタッフ個人所有端末での業務利用 |
| 紙併用モード | 紙→アプリ切替の移行期に、両運用を並行させる施設設定フラグ |
